1. Introduction

1.1 Purpose
This Software Requirements Specification (SRS) defines the complete functional
and non-functional requirements for ConcoursDoctor — an integrated web platform
designed to manage the full lifecycle of doctoral entrance examinations at Algerian
higher education institutions.
This document serves as the authoritative agreement between the development team
and the supervising faculty. It will be used as the primary reference throughout design,
implementation, testing, and validation.

1.2 Scope
The ConcoursDoctor platform covers the entire doctoral examination process from
the moment candidate data is imported to the moment the final results are officially
signed and published. Specifically, the system handles :
— Candidate importation from CSV/Excel files or via REST API from an existing
pre-registration system
— Examination planning : subjects, schedules, rooms, call lists
— Real-time attendance tracking by exam room supervisors via a mobile-friendly
interface
— Anonymization of examination copies using unique cryptographically generated
codes
— Double-blind correction with automatic discrepancy detection and third-corrector
arbitration
— Electronic deliberation including final ranking, coefficient-weighted averages, and
official PV generation
— Audit logging of every sensitive action throughout the process
Out of scope : The pre-registration platform, student academic records, and any
systems prior to the point of candidate list delivery.

1.3 Product Perspective
ConcoursDoctor is a standalone system designed to interface with existing university
information systems. It takes over after the pre-registration phase is complete.

1.3.1 System Interfaces
— Receives candidate data via REST API or CSV/Excel file import from the existing
pre-registration system
— Exports the final deliberation PV as a signed PDF document
— Integrates with the institutional email server (SMTP) for automated convocation
dispatch
1.3.2 User Interfaces
— Web application (React.js) for administrators, coordinators, correctors, and jury
members — accessible on PC and tablet
— Progressive Web App (PWA) optimized for tablet/smartphone use by exam
supervisors in examination rooms, with offline-first capability
1.3.3 Hardware Interfaces
— Document scanners : exam copies are scanned and uploaded ; the system supports
standard scanner output formats (PDF, JPEG)
— Optional : QR code / barcode readers for automatic copy-to-code association
1.3.4 Software Interfaces
— Frontend : React.js — a component-based JavaScript library for building the user
interface
— Backend : Django REST Framework (Python) exposing a RESTful API documented
via Swagger/OpenAPI
— Database : MySQL with strict role-based access ; nominative and anonymous data
stored in separate schemas
— Authentication : JWT (JSON Web Token) via Django REST Framework SimpleJWT
— Version control : Git / GitHub
— Project management : Trello
— Modeling and design : draw.io
1.3.5 Communication Interfaces
— HTTPS for all client-server communication (TLS 1.2 minimum)
— SMTP for email notifications (convocations, alerts)
— REST API for integration with the pre-registration system

1.6 Limitations
— The system does not manage the pre-registration of candidates ; it begins at the
point of receiving a validated candidate list
— Physical steps (envelope sealing, overnight custody by the Director, subject lottery
ceremony) are supported by PV generation but are not replaced by the software
— Subject Designers, the Department Head, and the Director are physical actors with
no system accounts ; their roles are recorded solely through generated PV documents
— Offline functionality is limited to the Supervisor PWA ; all other interfaces require
an active internet connection
— The platform is designed for Algerian doctoral examination regulations and may
require adaptation for other regulatory contexts
— The platform cannot prevent a candidate from writing identifying information (e.g.,
their name) directly on their exam copy. The physical anonymization procedure
must include an explicit instruction to candidates prohibiting this, enforced by
supervisors during the exam. Any copy found with visible identity information shall
be flagged by the Anonymity Commission and handled according to institutional
regulations

1.7 Assumptions and Dependencies
— The university’s hosting servers are operational, secured, and maintained by IT staff
— The data format from the pre-registration system is stable, documented, and accessible via API or structured file export
— Supervisors have access to a reliable Wi-Fi or 4G connection in examination rooms
— All system users have valid institutional email addresses
— Scanners for digitizing exam copies are available and produce standard output
formats (PDF, JPEG)

2.1.1 API Interface

CD-EI-06

The system shall accept scanned exam copy uploads in PDF and JPEG formats.
CD-EI-07

Optionally, the system shall support automatic QR code / barcode detection on
uploaded scan images to associate copies with their anonymous codes without
manual input.

2.2 Functional Requirements
2.2.1 Authentication and Access Control
CD-FR-AUTH-01

The system shall authenticate all users via email and password, issuing a JWT
token valid for 8 hours upon success.
CD-FR-AUTH-02

The system shall invalidate a user’s JWT token upon logout.
CD-FR-AUTH-03

After 3 consecutive failed login attempts, the system shall lock the account for 15
minutes and notify the System Administrator.
CD-FR-AUTH-04

The system shall enforce Role-Based Access Control (RBAC) with the following
roles : ADMIN, CFD_HEAD, COORDINATOR, CORRECTOR, SUPERVISOR, JURY_PRESIDENT,
JURY_MEMBER, ANONYMITY_COMMISSION. Subject Designers, the Department Head,
and the Director are physical actors and shall not have system accounts.
CD-FR-AUTH-05

Each role shall have a strictly defined set of permitted actions. No role shall be able
to perform actions outside its defined scope.

2.2.3 Examination Planning
CD-FR-EXAM-01

The Administrator shall be able to define one or more examination subjects, each
with a name, coefficient, passing score threshold, and maximum score.
CD-FR-EXAM-02

The Administrator shall be able to schedule each exam subject with a specific date,
start time, duration, and one or more assigned rooms.
CD-FR-EXAM-03

The system shall generate a printable call list (PDF) for each exam room, listing
candidates sorted by seat number, including their application number and full name.
CD-FR-EXAM-04

The call list shall also be accessible digitally by supervisors via the PWA interface.
CD-FR-EXAM-05

The system shall support a subject lottery process : the CFD Head can record
the lottery result (which of the 3 prepared subjects was selected) and generate the
corresponding PV.
2.2.4 Attendance Management
CD-FR-ATT-01

The Supervisor shall be able to mark each candidate as PRESENT or ABSENT
via a touch-friendly PWA interface during the exam.
CD-FR-ATT-02

The PWA interface shall function in offline mode, storing attendance data locally
and synchronizing automatically when internet connectivity is restored.
CD-FR-ATT-03

The Supervisor shall be able to undo an attendance marking for a candidate at any
time before the final attendance submission is confirmed.
CD-FR-ATT-04

The system shall display a real-time counter showing how many candidates have
been marked out of the total expected per room.
CD-FR-ATT-05
Any candidate marked ABSENT for a mandatory examination shall be automatically and
permanently eliminated from the competition. Their status shall be set to ELIMINATED
and they shall not appear in any further correction or deliberation steps.

2.2.5 Anonymization
CD-FR-ANON-01

Upon confirmation of attendance, the system shall automatically generate a unique
anonymous code for each PRESENT candidate in the format DOCT-YYYY-XXXX,
where YYYY is the exam year and XXXX is a zero-padded number derived from
a cryptographically secure pseudo-random number generator (CSPRNG). This
supports up to 9,999 candidates per session, which is sufficient for doctoral exam
scale.
CD-FR-ANON-02

The correspondence table linking a candidate’s identity to their anonymous code
shall be stored in encrypted form in a separate database schema. This table is
accessible only to the internal system process during the deliberation closure
operation, and to the ADMIN role for audit purposes. No other user role can query it
at any time.
CD-FR-ANON-03

The Anonymity Commission shall scan and upload exam copies. Each copy shall
be associated with its anonymous code either manually (by entering the code) or
automatically (via QR/barcode detection).
CD-FR-ANON-04

Once the association between a copy and its anonymous code is confirmed, the
system shall hide all candidate identity information from all correction-phase
interfaces. Only the anonymous code shall be visible to correctors and coordinators.
CD-FR-ANON-05

The system shall generate a PV of Anonymization recording the number of copies
coded and the timestamp of the operation, signed by the Anonymity Commission
members.

2.2.6 Double Correction Management
CD-FR-COR-01

The Coordinator shall assign each anonymized copy to exactly two Correctors for
independent blind correction.
CD-FR-COR-02

Each Corrector shall only see the copies assigned to them, identified solely by their
anonymous code.
CD-FR-COR-03

Each Corrector shall enter a numerical grade for each assigned copy. The interface
shall enforce the configured score range (0 to maximum score).
CD-FR-COR-04

Once both grades are entered for a copy, the system shall automatically calculate
the absolute difference between them.
CD-FR-COR-05

If the difference exceeds the configured discrepancy threshold (default : 3 points),
the system shall immediately trigger an alert to the Coordinator.
CD-FR-COR-06

The Coordinator shall be able to designate a third Corrector to arbitrate discrepant copies. The third Corrector’s interface shall display both existing grades
anonymously before they enter their own grade.
CD-FR-COR-07

The final grade for a copy shall be computed according to the configured rule :
AVERAGE (of two grades), MEDIAN (of three grades), or THIRD_CORRECTOR (use only
the third corrector’s grade). This rule is configurable per exam subject.
CD-FR-COR-08

Once the Coordinator validates the final grades for an exam subject, no Corrector
shall be able to modify any grade for that subject.
CD-FR-COR-09

The system shall generate a PV of Correction recording all final grades, the number
of discrepancies encountered, and the arbitration decisions made.

2.2.7 Electronic Deliberation
CD-FR-DEL-01

The deliberation phase shall be accessible only to users with roles JURY_PRESIDENT
and JURY_MEMBER, and only after all exam subjects have been validated by the
Coordinator.
CD-FR-DEL-02

The system shall automatically compute each candidate’s weighted average using
the configured coefficients per subject.
CD-FR-DEL-03

The system shall generate a provisional ranking of all eligible candidates, sorted
by descending weighted average. The ranking shall display only anonymous codes
until deliberation is closed.
CD-FR-DEL-04

The system shall apply the configured admissibility threshold (e.g., minimum
average of 10/20) and automatically flag candidates as ADMITTED, WAITING_LIST,
or REJECTED.
CD-FR-DEL-05

The Jury President shall be able to mark the deliberation as closed. This action
shall be irreversible without Administrator intervention.
CD-FR-DEL-06

Upon closure of deliberation, the internal system process shall automatically lift
anonymity by replacing all anonymous codes in the results with the corresponding
candidate identities, using the encrypted correspondence table. This operation is
performed by the system itself ; no human user directly accesses the correspondence
table during this step.
CD-FR-DEL-07

The system shall generate the official PV of Deliberation in PDF format, including :
full ranked results with real identities, admitted candidates list, waiting list, rejected
candidates list, and timestamp of deliberation closure.
CD-FR-DEL-08

The system shall support optional electronic signatures on the PV by the Jury
President and Jury Members.
CD-FR-DEL-09

All deliberation data (grades, PV, logs) shall be archived with a timestamp and
shall be immutable after deliberation closure.

2.2.9 Audit Logging
CD-FR-LOG-01

The system shall maintain an immutable audit log recording every sensitive action
performed by any user.
CD-FR-LOG-02

Each log entry shall capture : timestamp, user ID, user role, IP address, action
type, and the affected object (e.g., copy ID, candidate ID).
CD-FR-LOG-03

Actions that must be logged include (but are not limited to) : login/logout, grade
entry, grade modification, identity access, deliberation closure, PV generation, and
any Administrator configuration change.
CD-FR-LOG-04

The audit log shall be accessible only to the System Administrator and shall not be
editable or deletable by any user.

CD-NFR-ERG-01

The React.js web interface shall be fully responsive, functioning correctly on screens
with widths from 320px (smartphones) to 2560px (large monitors).
CD-NFR-ERG-02

The Supervisor PWA shall allow a trained supervisor to mark attendance for 30
candidates in under 3 minutes under normal conditions.
CD-NFR-ERG-03

All form validation errors shall be displayed inline, adjacent to the problematic
field, with a clear and human-readable message.
CD-NFR-ERG-04

Every destructive or irreversible action (e.g., eliminating a candidate, closing
deliberation) shall require a confirmation dialog explicitly stating the consequences.
CD-NFR-ERG-05

The Corrector interface shall display the scanned copy and the grade input field on
the same screen without requiring horizontal scrolling.

CD-NFR-PERF-01

The Supervisor PWA shall fully load within 2 seconds on a 3G mobile connection
(estimated bandwidth : 3 Mbps).
CD-NFR-PERF-02

All API responses shall complete within 500ms under normal load conditions (fewer
than 200 concurrent users).
CD-NFR-PERF-03

The system shall support at least 500 concurrent authenticated users without
degradation in response time.
CD-NFR-PERF-04

PDF generation for PVs and result documents shall complete within 10 seconds
regardless of the number of candidates.

2.6 Security Requirements
CD-NFR-SEC-01

All data in transit shall be encrypted using TLS 1.2 or higher. HTTP connections
shall be automatically redirected to HTTPS.
CD-NFR-SEC-02

All user passwords shall be hashed using bcrypt with a minimum cost factor of 12
before storage.
CD-NFR-SEC-03

JWT tokens shall expire after 8 hours. Refresh tokens shall expire after 24 hours.
CD-NFR-SEC-04

The identity-to-anonymous-code correspondence table shall be stored in a separate
MySQL schema, accessible only to the internal system process during deliberation
closure and to the ADMIN role for audit purposes. No other user role may query this
table at any time.
CD-NFR-SEC-05

The correction server shall have no database-level access to the nominative candidate
table.
CD-NFR-SEC-06

Every access to a candidate’s identity, every grade modification, and every configuration change shall generate an immutable log entry as defined in [CD-FR-LOG-01].
CD-NFR-SEC-07

Once grades are validated by the Coordinator, they shall be immutable. Any
modification attempt shall be blocked at the API level and logged.
CD-NFR-SEC-08

The system shall implement OWASP Top 10 protections including SQL injection
prevention, XSS sanitization, and CSRF protection on all state-changing endpoints.
CD-NFR-SEC-09

The database shall be automatically backed up every 24 hours. Backups shall be
encrypted and stored separately from the main server.
